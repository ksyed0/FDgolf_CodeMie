# US-0002 — Supabase Project Provisioning

**Epic:** EPIC-0001 Foundation & Infrastructure
**Status:** Design (in progress)
**Author:** Compass
**Depends on:** US-0001

---

## 1. Goal

With the Next.js scaffold in place, this story provisions the Supabase backend so all subsequent stories have a live PostgreSQL database and an auth provider to connect to. The deliverable is a running Supabase project (cloud or local via `supabase start`) with email/password auth enabled, connection credentials documented in `.env.local`, and the Supabase CLI config committed to the repo for reproducible local dev.

## 2. Acceptance Criteria

- AC-0007: Supabase project created (cloud or local via `supabase start`)
- AC-0008: Email + password auth provider enabled in Supabase Auth settings
- AC-0009: Project URL and anon key documented in `fdgolf-app/.env.local` (not committed; only `.env.local.example` is committed)
- AC-0010: `supabase/config.toml` checked into repo under `fdgolf-app/supabase/config.toml` for reproducible local dev

## 3. Out of Scope

- Database schema or migrations (US-0005)
- RLS policies (US-0006)
- Service role key usage (used only server-side in later stories)
- OAuth providers (Phase 2)
- 2FA configuration (EPIC-0010, v1.1)
- Supabase Storage bucket setup (needed later by EPIC-0002 and EPIC-0008)

## 4. Notes

- **Local vs cloud:** Both are valid. For CI reproducibility, `supabase start` is preferred during development. Production will use the cloud project.
- **Security:** `SUPABASE_SERVICE_ROLE_KEY` must never be placed in `NEXT_PUBLIC_*` env vars. AC-0009 documents the anon key (public) only in `.env.local`; the service role key stays in server-only env.

## Technical Design

**Author:** Keystone

### Supabase CLI setup

The Supabase CLI project lives at `fdgolf-app/supabase/`. This colocation keeps database migrations alongside the app code they serve.

```
fdgolf-app/
  supabase/
    config.toml       ← committed; defines project_id, auth settings, etc.
    migrations/       ← empty for now; US-0005 adds schema migrations
    seed.sql          ← empty for now; US-0008 adds club seed
  .env.local          ← gitignored; holds NEXT_PUBLIC_SUPABASE_URL + keys
```

**`fdgolf-app/supabase/config.toml`** (key settings for auth):
```toml
[auth]
enabled = true
site_url = "http://localhost:3000"
additional_redirect_urls = []

[auth.email]
enable_signup = true
double_confirm_changes = true
enable_confirmations = false   # disabled for Phase 1 dev speed; enable in prod
```

### Environment variable wiring

After provisioning, the developer copies `.env.local.example` to `.env.local` and fills in:

```bash
NEXT_PUBLIC_SUPABASE_URL=http://127.0.0.1:54321          # local
NEXT_PUBLIC_SUPABASE_ANON_KEY=<local-anon-key>
SUPABASE_SERVICE_ROLE_KEY=<local-service-role-key>
NEXT_PUBLIC_MAPBOX_TOKEN=<from Mapbox dashboard>
```

For cloud: replace the URL and keys with the cloud project values from the Supabase dashboard.

### Gitignore additions

The root `.gitignore` already excludes `fdgolf-app/.env.local`. No additional entries needed for Supabase; `supabase/.branches/` and `supabase/.temp/` are auto-ignored by the Supabase CLI's own `.gitignore` generation.
