# US-0006 — Row Level Security Policies Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a new migration file `supabase/migrations/20260609000001_rls_policies.sql` that enables RLS on all 16 tables created in US-0005, defines role-aware access policies, creates the `public_hole_scores` view, and is proven safe by a `validate-rls.sh` test script — all without touching the initial schema migration.

**Architecture overview:**

- RLS enable + all policy DDL lives entirely in `20260609000001_rls_policies.sql`.
- Role checking uses the `user_roles` table via subquery: `EXISTS (SELECT 1 FROM user_roles WHERE player_id = auth.uid() AND role = 'admin')` for global admin bypass; `EXISTS (SELECT 1 FROM user_roles WHERE player_id = auth.uid() AND role = 'tournament_organizer' AND tournament_id = <col>)` for scoped organizer access.
- `players.is_admin` is a legacy convenience column but is NOT the authoritative role check; `user_roles` is authoritative.
- Player identity: `auth.uid()` returns the UUID of the currently authenticated Supabase Auth user.
- The public leaderboard privacy requirement (AC-0033) is delivered as a Postgres VIEW `public.public_hole_scores` that joins `team_hole_scores` to `players` but selects only safe columns (no email, phone, year_of_birth, gender). The view is SELECT-only; inserts/updates go through the base tables.
- `shot_edits` is insert-via-trigger only — the player-facing RLS blocks direct INSERT; admin and service role can insert directly as an escape hatch.
- `supabase db reset` with both migration files must succeed from a cold start.

**Executed by:** Relay (DevOps agent) — this story is pure database work. No files under `fdgolf-app/src/`, `fdgolf-app/app/`, or `fdgolf-app/components/` are touched.

- [ ] **Step 2: Append the helper functions**

  Append:

  ```sql
  -- ============================================================
  -- 2. Role-check helper functions
  --    SECURITY DEFINER so the user_roles lookup is privileged.
  --    STABLE because results don't change within a transaction.
  -- ============================================================

  -- Returns TRUE when the calling user has role='admin' in user_roles
  -- (tournament_id is NULL for global admin rows).
  CREATE OR REPLACE FUNCTION fdgolf_is_admin()
  RETURNS BOOLEAN
  LANGUAGE sql
  STABLE
  SECURITY DEFINER
  AS $$
    SELECT EXISTS (
      SELECT 1
      FROM   user_roles
      WHERE  player_id     = auth.uid()
        AND  role          = 'admin'
        AND  tournament_id IS NULL
    );
  $$;

  -- Returns TRUE when the calling user is a tournament_organizer
  -- for the given tournament_id.
  CREATE OR REPLACE FUNCTION fdgolf_is_organizer_for(p_tournament_id uuid)
  RETURNS BOOLEAN
  LANGUAGE sql
  STABLE
  SECURITY DEFINER
  AS $$
    SELECT EXISTS (
      SELECT 1
      FROM   user_roles
      WHERE  player_id     = auth.uid()
        AND  role          = 'tournament_organizer'
        AND  tournament_id = p_tournament_id
    );
  $$;
  ```

  > **Design note:** `SECURITY DEFINER` is used so that the subquery inside reads `user_roles` using the function owner's privileges — preventing a bootstrapping problem where the player's RLS policy on `user_roles` would otherwise block the lookup needed to evaluate their own policy.

  --     Joins team_hole_scores with teams and players (contributing player)

### TASK-0303: Write validate-rls.sh and run full verification (AC-0034)

**Files:**
- Create: `Claude/fdgolf-app/supabase/validate-rls.sh`

AC-0034 requires SQL tests that verify each policy using a non-privileged JWT. The approach: a `validate-rls.sh` bash script that uses `psql` with `SET LOCAL role` and `SET LOCAL "request.jwt.claims"` (Supabase's mechanism for injecting `auth.uid()` into a local transaction) to simulate a player making queries.

**How Supabase local JWT simulation works:**

In the local Supabase stack, `auth.uid()` is derived from the `request.jwt.claims` GUC:
```sql
SET LOCAL "request.jwt.claims" = '{"sub": "<uuid>", "role": "authenticated"}';
SET LOCAL ROLE authenticated;
```
This makes `auth.uid()` return the specified UUID for the duration of the transaction. Running this inside a `BEGIN`/`ROLLBACK` block leaves no data.
