Overview
App Structure & Roles
FDgolf is a Progressive Web App (PWA) — no app store required. Players open it in their phone browser on tournament day. Four distinct roles, each with a tailored experience.

⛳ Player

The core user. Registers before tournament day, joins a team, and uses the scoring interface during the round.

  • Self-registers with name, company, email
  • Assigned to a team by organizer or admin
  • Scores shots on phone during round
  • Views live leaderboard

🎯 Tournament Organizer

A "concierge" role for bulk registering players and managing team assignments before the event.

  • Registers players on their behalf
  • Assigns players to teams
  • Sets team captains
  • Cannot override scores

🔧 Admin

Full tournament control. Pre-event setup and real-time management during the round.

  • Creates & activates tournament
  • Manages holes, clubs, sponsors
  • Overrides scores if disputed
  • Completes the tournament

👥 Public / Spectator

No login required. The live leaderboard at /live/cibc-granite-ridge-2026 is accessible to anyone with the link.

  • Real-time leaderboard (no auth)
  • Sponsor banner visible
  • Auto-refreshes every 30 seconds
Technical Architecture
Key Design Decisions

Offline-First Write Queue

Every shot is written to localStorage first, then synced to Supabase. If the player loses signal mid-round, shots accumulate locally and auto-flush when connectivity returns. Up to 5 retry attempts per shot.

Best Ball Calculation

Calculated server-side via a Supabase Edge Function after every score save. The edge function finds the minimum stroke per team+hole and marks is_best_ball = true on the winning player's score row.

Live Leaderboard (125 clients)

Supabase Realtime subscription on the scores table with a 5-second client-side debounce. On any score change event, the client waits 5s before re-fetching — preventing a 125-client storm on every score save.

Google Maps Caching

The Google Maps instance is cached in a React ref (useRef) so it's only created once per page load. ~18,000 map loads for a full tournament (125 players × 8 views/hole × 18 holes) — within free tier of 28K/mo.

Variable Team Size

Teams support 2–6 players (max_players column). Default is 4. Enables both standard foursomes and smaller/larger groups as needed for this event's headcount.

No App Store Required

Players open a URL on mobile Safari / Chrome. PWA manifest allows "Add to Home Screen" for native-like experience. The main risk is iOS Safari's PWA offline support — see Open Questions.

Check-in Tablet Flow

Organizer uses a tablet at the registration desk. As players arrive, they're registered (creating a Supabase auth account with a temp password) and placed on their team. Players get a "reset password" email to set their own password before teeing off.

Captain Field

Each team has one designated captain (set via teams.captain_id). The captain is responsible for confirming the score at end of hole. This is a UX label only — all team members can record shots.

Variable Team Size

Organizer sets max_players (2–6) per team. The team card shows this limit and prevents over-assignment. Default is 4 for Best Ball foursomes.

One Phone, Four Players

One player acts as scorekeeper for the group. They tap the player pill to switch who they're recording for. No need for each player to have their own phone open during the round.

GPS Captured at Shot Time

The phone's GPS coordinates are captured when the player taps an outcome button. These appear as shot markers on the hole map. The "distance to pin" is calculated in real-time using the Haversine formula.

Best Ball Auto-Calculated

When "Sunk!" is tapped for any player, the edge function runs and marks is_best_ball = true on the minimum stroke score for that team+hole. The hole summary card reflects this immediately.

Public URL Use Cases

The /live/[slug] URL can be displayed on a TV in the clubhouse, shared in the event WhatsApp group, or projected at the 19th hole. No account needed — just the link.

ISR (30-second refresh)

The public page uses Next.js Incremental Static Regeneration with revalidate = 30. It serves cached HTML (fast load, low DB cost) and regenerates every 30s as new scores arrive.

Audit Trail

The scores table has override_by (admin player ID) and override_at (timestamp) columns. Overrides are visible in the admin table with a distinct visual treatment.

Best Ball Recalc

After an admin override, the best ball edge function is called again for that team+hole. The minimum stroke changes if the override affects who had the best score on that hole.

localStorage Queue

Shots are written to localStorage['fdgolf_sync_queue'] first — before any network attempt. If the page is refreshed mid-queue, shots are not lost.

Max 5 Retries

Each queued shot retries up to 5 times on flush. If it fails 5 times (e.g. server error, not just network), it's dropped and the player should contact the admin for a manual override.

iOS PWA Risk

iOS Safari PWA storage can be cleared by the OS if the device is low on storage. For tournament day, recommend players do NOT delete the PWA and have at least 200MB free. Fallback: use Safari directly (not the home screen app).

Open Questions
Items for PO Review
These items need your input before or during build. Priority: High = blocks development, Medium = needed before launch, Low = can decide day-of.
  • HIGH
    Score attestation: Should the team captain be required to confirm/sign off on each hole score before it's submitted? Or is the "Sunk!" button tap sufficient? This affects the UX flow and data model.
  • HIGH
    Concierge registration flow: When the organizer registers a player on their behalf, what password should be set? Options: (a) temp password emailed to player, (b) player completes registration via magic link, (c) organizer sets password verbally at check-in.
  • MED
    Weather / delay pause: Is there a need for a "pause tournament" state? (e.g. thunder delay). Currently the only states are setup → active → completed. A "paused" state could prevent scoring during a delay.
  • MED
    Tournament archive: After June 22, should this year's scores be preserved and accessible? Do you want a tournament history view? This affects whether we need a "past tournaments" feature or if the app is one-and-done.
  • MED
    Hole-by-hole scorecard: The leaderboard shows total score vs par. Do players/admins also need a per-hole scorecard view (stroke per hole, best ball highlighted per hole)?
  • LOW
    Team naming: Teams currently have a team_name field. Should teams be named (e.g. after the captain's company)? Or stay as "Team 1", "Team 2"? This affects leaderboard display.
  • LOW
    Mulligan tracking: Mulligans are currently treated as a shot outcome (recorded but not counted in score). Do they need to be tracked for any reporting purpose? Or is logging them sufficient?
Constraints & Risks
Known Risks

iOS Safari PWA Offline

iOS Safari has inconsistent PWA offline support compared to Chrome. The localStorage queue works, but Service Worker caching (for full offline app shell) is less reliable. Mitigation: test on iPhone before tournament day.

125 Concurrent Users

Supabase Realtime handles 125 concurrent subscriptions well, but the 5s debounce is critical. Without it, every score save from every player would trigger a DB fetch on all 125 clients simultaneously.

GPS Accuracy on Course

Hole GPS coordinates in seed data are approximate (based on Granite Ridge course center). Actual pin locations may differ by 30–100m. Admin should update pin_lat/pin_lng via the holes admin page before the round.

Shotgun Start (All Teams, Same Time)

All teams start simultaneously on different holes. The round_state table tracks current_hole per team independently. There's no dependency between teams — each team progresses at their own pace around the course.

No App Store Dependency

A web app avoids App Store review delays and the friction of asking 125 players to download an app. Players open a URL — that's it. The tradeoff is slightly worse offline support vs a native app.
